# 06. Argus UI 设计规范

## 目标

本文档定义 Argus 前端界面的长期设计语言。后续新增页面、重构组件或由 AI 生成 UI 代码前，必须先阅读本规范，并优先遵守这里定义的产品气质、token、组件语义与代码落地规则。

Argus 不是通用后台模板，也不是演示站点。它应呈现为专业金融 AI 操作终端：冷静、克制、锐利、可靠，同时在关键数据与 Agent 状态上具备大胆且清晰的视觉表达。

## 产品气质

- **专业金融终端**：界面应像交易、风控、研究和 Agent 调度的控制台，而不是营销落地页。
- **极简但不单薄**：减少装饰，用分割线、栅格、数字层级和状态色建立秩序。
- **锐利且大方**：默认低圆角或直角，避免软萌、卡通、胶囊化风格。
- **大胆但标准**：关键指标可以强对比、强字号；基础组件必须一致、可预测、可复用。
- **细腻但克制**：允许细微背景层级、边框透明度、hover/focus 状态；避免廉价阴影、厚重渐变和过度动效。

## 反目标

以下视觉和实现倾向应避免：

- 大面积 `rounded-xl`、`rounded-2xl`、`rounded-full` 造成的模板感。
- 多套按钮、卡片、Badge、弹窗标题样式并存。
- 依赖大阴影、玻璃拟态、强模糊或彩色渐变制造“高级感”。
- 间距松散、标题层级随意、关键数字不突出。
- 将交易图表、Agent 状态、配置表单做成彼此无关的界面。
- 删除现有 DOM id、事件名或 bridge 契约来换取视觉简洁。

## 视觉原则

### 暗色优先

Argus 默认运行在暗色环境。浅色 token 可保留，但所有核心体验必须以 `.dark` 为准校准。

背景层级建议：

| 层级 | 用途 |
| --- | --- |
| `--background` | 应用最底层画布 |
| `--card` | 面板、弹窗、图表卡片 |
| `--popover` | 浮层、菜单、Dialog |
| `--muted` | 次级区域、表单填充、列表 hover |
| `--border` | 默认细分割线 |

### 低圆角

- 全局基础半径应接近 `0.25rem`。
- 面板、卡片、弹窗默认使用 `rounded-sm` 或 `rounded-md`。
- 只有小型状态点、头像、图标按钮中装饰性圆点可以使用 `rounded-full`。
- 不要用大圆角表达“现代”，Argus 的现代感来自秩序、对比和信息密度。

### 强栅格与细分割线

- 主界面以明确的垂直/水平分割组织信息。
- 优先使用 `border-border/70`、`border-border/80`、`divide-border`、`ring-1 ring-border/70`。
- 阴影只作为浮层层级提示，不作为卡片默认语言。

### 高信息密度

- 工具栏高度建议 36-44px。
- 表单控件高度建议 28-34px。
- 指标卡允许紧凑，但标签、数值、单位和状态必须清楚。
- Dashboard 可以更大胆，但不应牺牲扫描效率。

### 关键数字优先

- 资金、收益、胜率、状态、Agent 运行态是一级信息。
- 关键数字使用更大字号、紧字距、清晰正负色。
- 说明文字、来源、时间戳退到 muted 层级。

## 设计 token 规则

### 色彩

- 基础色以中性黑灰为主，避免蓝紫渐变模板感。
- 主强调色用于“当前选中 / 核心控制 / Agent 激活”，不应滥用。
- 状态色语义固定：
  - 盈利、在线、运行：emerald
  - 风险、亏损、错误：red/rose
  - 暂停、等待、警告：amber
  - 信息、系统、截图：cyan/sky
  - 中性、未知、未配置：muted/foreground

### 文字

- 标题：短、克制、字重 600 左右。
- 标签：使用小字号与较宽 tracking，适合终端感。
- 正文：保持 12-14px，可读优先。
- 数字：可使用 24-40px，`tracking-tight`，避免过粗。

### 半径

- `--radius` 作为全局源头，默认不超过 `0.25rem`。
- `rounded-md` 是大多数组件上限。
- 新增 `rounded-xl/2xl/full` 前必须有明确理由。

### 阴影

- 默认面板和卡片不使用阴影。
- Dialog、SelectContent 等浮层可使用轻量阴影配合 ring。
- 不使用彩色投影作为品牌效果。

## 组件规范

### Button

- 默认按钮要干净、紧凑、边界清楚。
- 主要操作使用高对比实色；次要操作使用 outline 或 ghost。
- 图标按钮必须有 `aria-label`。
- 不同页面不要自行发明按钮圆角、尺寸和 hover。

### Badge / Status

- Badge 用于状态，不用于装饰。
- 状态色必须与语义一致。
- 避免大胶囊；默认小半径、细边框、紧凑字号。

### Card / Panel

- Card 是信息容器，不是装饰卡片。
- 默认无大阴影，依靠背景层级和细边框。
- Panel 需有清晰 header、body、必要时 footer。
- Panel header 中的标题、状态、操作按钮顺序保持一致。

### Dialog / Modal

- 弹窗统一由 Radix/shadcn Dialog 承载。
- DialogContent 默认低圆角、强边框、清晰 header 分割线。
- 关闭按钮样式统一，但必须保留已有 id。
- 弹窗 body 必须管理滚动，不让 header/footer 被滚走。
- 所有 Dialog 必须有可访问标题和描述；视觉隐藏描述可用 `sr-only`。

### Metric

- Metric 由 label、value、辅助说明、可选趋势组成。
- label 小写/大写风格需统一，建议使用 `tracking-[0.14em] uppercase`。
- value 是视觉中心，不要被图标或装饰抢占。

### Form

- 表单以扫描效率为核心。
- Label、输入、说明、错误信息层级必须稳定。
- 输入框使用低圆角、细边框、暗色填充，不用大块浅色背景。

### Empty / Loading / Error

- 空状态要直接说明下一步动作。
- Loading 不应大面积占屏；优先局部状态。
- Error 文案应可执行，避免只显示异常对象。

## 页面规范

### 主工作台

- TradingView 图表是主画布，Argus 面板是右侧命令面板。
- 图表外壳只提供必要控制、布局分割和状态，不抢图表注意力。
- 右侧面板折叠/展开必须保持平滑且不影响图表运行。

### Argus 面板

- 定位为 Agent command panel，而不是聊天玩具。
- 顶部必须呈现 Agent 品牌、运行状态、全局开关。
- 持仓、订单、会话历史必须有明确层级。
- 新增会话项、状态块时复用统一 panel/metric/status 组件。

### Dashboard

- Dashboard 是运营驾驶舱，不是普通表单弹窗。
- 首屏应优先展示资金、盈亏、胜率、策略状态。
- 统计区间和配置操作应清晰但退居二级。
- 图表和指标需要统一边框、背景、字号和状态色。

### 其他弹窗

- 配置中心、策略中心、交易复盘、LLM 详情和图表预览共享弹窗外壳。
- 不同弹窗允许内容布局不同，但 header、关闭按钮、body padding、footer 操作规则一致。

## 交互规范

- hover 只改变轻微背景或边框，不制造跳跃。
- focus 必须清晰可见，继续使用 ring token。
- active 可有轻微位移，但不要在复杂控件中造成布局抖动。
- 动效时长建议 100-200ms。
- 滚动区域需清晰；避免整页和内部区域互相抢滚动。

## 代码落地规则

- 优先使用 Tailwind v4 utility、shadcn/Radix primitives 和 CSS variables。
- 新增文件名使用 kebab-case，React 组件使用 PascalCase。
- UI 不得直接导入 `internal/*`。
- 不新增远程字体、CDN、第三方 UI 框架，避免触碰 CSP。
- 不破坏 `window.argus`、RPC/WS、TradingView 初始化逻辑。
- 必须保留 `argus-renderer.ts` 或自动化依赖的 DOM id、事件名和挂载点。
- 新增导出组件应有 JSDoc，说明组件用途和约束。
- 视觉常量优先进入 CSS variables 或共享组件，不在页面里重复硬编码。

## AI 修改 UI 前检查清单

1. 已阅读本文档和同目录重构规范。
2. 明确本次改动属于 token、primitive、app shell、页面组件还是遗留 CSS。
3. 没有新增大圆角、廉价渐变、随意阴影或独立风格按钮。
4. 没有删除 DOM id、事件名或 bridge 契约。
5. 新增状态色符合语义。
6. Dialog、Panel、Metric 优先复用共享外壳。
7. 已运行类型检查和 lint，或在结果中说明无法运行的原因。
